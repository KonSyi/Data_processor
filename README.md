# Техническое задание: DataProcessor

## 1. Назначение

Приложение для обработки данных экспериментальной установки. Превращает сырые
записи импульсов (shots) в «паспорт импульса» — набор рассчитанных характеристик
(числа) и графиков по выбранным диагностикам.

Эволюция: из просмотрщика TDMS → в инструмент обработки сигналов с сохраняемыми
рецептами расчёта и (в перспективе) пакетной обработкой.

---

## 2. Предметная область

### 2.1. Импульс (Shot)
Центральная сущность системы. Один импульс соответствует одному выстрелу установки.

**Состав одного импульса:**
| Источник              | Содержимое                                              |
|-----------------------|---------------------------------------------------------|
| TDMS-файл (~123 МБ)   | Быстрые осциллограммы (~5 секунд), много групп/каналов   |
| `.tdms_index`         | Служебный индекс nptdms (отдельно не обрабатывается)     |
| `_Настройки_*` (INI)  | Параметры АЦП и человеческие имена каналов               |
| Отрезок дневного CSV  | Медленные датчики, вырезается по времени импульса         |

**Идентификация импульса:**
- ID импульса: `<номер>` + `<время начала>` из имени файла
  (пример: `27856__12_14_15` → номер `27856`, время `12:14:15`).
- Дата: из имени папки-дня.
- Полный уникальный ключ: `дата + номер` (либо `дата + время`).

**Объём:** до ~100 импульсов в одной папке-дне.

### 2.2. Файл настроек (`_Настройки_*`)
INI-подобный формат, секции на русском, числа с запятой как разделителем.

Содержит:
- Параметры АЦП (слотов): частота дискретизации, число отсчётов, цифровой
  фильтр (вкл/выкл, порядок, частота среза), режим работы.
- Параметры каждого канала: физический канал (`PXI1Slot4/ai0`),
  **человеческое имя** (`MWI`), единицы измерения, входной диапазон.

**Назначение в системе:**
- Источник человеческих имён каналов (для отображения и маппинга диагностик).
- Источник метаданных АЦП (частота дискретизации → ось времени, калибровки).

### 2.3. Диагностика (Diagnostic)
Прикладная сущность над сырыми каналами. Может собираться из нескольких
каналов (возможно из разных групп TDMS) и/или из колонок CSV.

- Соответствие «диагностика → каналы» определяется именами каналов из настроек
  (детали маппинга — см. открытый вопрос O1).
- Корректность записи диагностики оценивается **пользователем визуально**
  (автоматической валидации на старте нет).

### 2.4. Метод обработки (Method)
Алгоритм, превращающий данные диагностики в результат.

**Свойства методов:**
- Привязаны к типу диагностики, но **выбираются пользователем** вручную.
- Одну диагностику можно обработать несколькими методами.
- Один метод может давать **несколько результатов** (числа + графики).
  Пример: «ВАХ зонда» → температура T + плотность n + график фита.
- Имеют **настраиваемые параметры** (пример: предварительный фильтр вкл/выкл +
  частота среза; границы интегрирования; диапазон фита).
- Источник данных: TDMS-каналы и/или колонки CSV.
- **Поддерживают цепочки** (зависимости): результат одного метода может быть
  входом другого (пример: RPA использует результат обработки зонда).
  Зависимости простые (не произвольный граф) — достаточно расчёта в порядке
  зависимостей.

### 2.5. Результат обработки (ProcessingResult)
- Скалярные величины (числа с единицами измерения).
- Графики — хранятся как **числовые данные (x, y) + метаданные отображения**,
  перерисовываются на лету (PNG не хранятся).
- Содержит информацию «как получено»: метод + параметры расчёта.

## 3. Функциональные требования

### 3.1. Просмотр (окно браузера импульсов)
- Загрузка импульса из папки (TDMS + настройки + соответствующий CSV-отрезок).
- Дерево диагностик/каналов с человеческими именами из настроек.
- Просмотр сырых осциллограмм для визуальной оценки корректности записи.
- Выбор каналов/диагностик для обработки.

### 3.2. Обработка
- Выбор метода(ов) для каждой выбранной диагностики.
- Настройка параметров метода перед расчётом.
- Расчёт с учётом зависимостей между методами (правильный порядок).
- Предварительная обработка (опциональный фильтр) перед основным методом.

### 3.3. Результаты (окно результатов)
- Отображение графиков и чисел.
- Экспорт всех результатов в CSV.

### 3.4. Сохранение сессии / кэш
- Результаты импульса сохраняются в **JSON-файл-рецепт** рядом с импульсом:
  - какие диагностики какими методами и с какими параметрами обработаны;
  - рассчитанные числа;
  - числовые данные графиков.
- Повторное открытие импульса загружает результаты из кэша **без пересчёта**.
- Формат должен быть пригоден для будущей пакетной обработки.

### 3.5. Пакетная обработка (будущее, заложить совместимость)
- Применение метода к множеству импульсов → сводная таблица.
- Кэш-рецепт должен позволять автоматический пересчёт без GUI.

## 4. Нефункциональные требования

| Требование        | Описание                                                       |
|-------------------|----------------------------------------------------------------|
| Тестируемость     | Слои `domain`, `processing`, `services` тестируются без Qt и    |
|                   | без реальных файлов (numpy-массивы на входе).                  |
| Отзывчивость UI   | Тяжёлые операции (чтение ~123 МБ TDMS, расчёт) не блокируют GUI. |
| Расширяемость     | Новый метод обработки добавляется без правки существующих.      |
| Локализация       | Русские подписи/имена — в слое представления, не в домене.      |
| Формат данных     | Учитывать запятую как десятичный разделитель в настройках/CSV. |

## 5. Архитектура (целевая структура)

```
DataProcessor/
├── domain/              # чистые сущности — ноль Qt, ноль I/O
│   ├── shot.py          # Shot: id, дата, пути к источникам
│   ├── channel.py       # Channel: сырьё (x, y, units) без форматирования
│   ├── diagnostic.py    # Diagnostic: маппинг каналов
│   └── result.py        # ProcessingResult: числа + данные графиков
│
├── io/                  # всё чтение/запись
│   ├── tdms_reader.py
│   ├── csv_reader.py        # + вырезание отрезка по времени
│   ├── settings_reader.py   # парсер Настройки* (INI, запятая, кодировка)
│   └── cache_store.py       # JSON-рецепт: load/save
│
├── processing/          # методы как объекты + реестр
│   ├── base.py          # интерфейс Method (входы/выходы/параметры)
│   ├── registry.py      # реестр доступных методов
│   └── methods/         # конкретные методы (rogowski, probe_iv, rpa, cmdt...)
│
├── services/            # оркестрация
│   ├── shot_loader.py       # собрать Shot из папки
│   └── processing_service.py# применить методы с учётом зависимостей
│
├── presentation/        # форматирование для UI (имена, числа, подписи осей)
│   └── formatters.py
│
├── ui/                  # только виджеты
│   ├── shot_browser/
│   └── results_window/
│
├── controller/          # тонкие координаторы
└── app/
└── main.py          # composition root (сборка зависимостей)

```

**Ключевые принципы:**
1. `domain`, `processing`, `services` **не импортируют Qt и не импортируют
   nptdms/чтение файлов напрямую**. Это основа тестируемости.
2. `domain` хранит **сырьё** (значения, единицы как есть). Форматирование строк,
   русские подписи, `dict[str, str]` — в `presentation`.
3. Контроллер — тонкий: «получил сигнал UI → вызвал service → отдал результат во
   View». Никаких вычислений и форматирования в контроллере.
4. `io` — единственный слой, знающий о nptdms, путях, кодировках, форматах файлов.

## 6. Открытые вопросы (решить до реализации)

- **O1. Маппинг диагностик.** Как именно «человеческое имя канала» (`MWI`) из
  настроек связывается с диагностикой и методом? Это:
  - (а) фиксированный конфиг установки «диагностика X = каналы [...] + метод Y»?
  - (б) пользователь собирает вручную каждый раз?
  - (в) выводится из соглашения об именах?

- **O2. Калибровки.** Нужны ли для расчётов калибровочные коэффициенты
  (например, пояс Роговского, делители)? Где они хранятся — в настройках, в
  отдельном конфиге, задаются в UI? (Влияет на параметры методов.)

- **O3. Ось времени / единицы.** Ось времени канала строится из частоты
  дискретизации настроек, или есть waveform-свойства в самом TDMS
  (`wf_increment`)? Что приоритетнее?

- **O4. Привязка CSV.** Достаточно ли «время начала TDMS + 5 сек», или
  длительность импульса переменная и её надо брать из данных?

- **O5. Идентичность диагностик между импульсами.** Набор каналов/диагностик
  одинаков для всех импульсов дня или может меняться от выстрела к выстрелу?
